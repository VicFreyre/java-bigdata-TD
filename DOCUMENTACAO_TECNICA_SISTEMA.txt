DOCUMENTAÇÃO TÉCNICA DO SISTEMA DE MATRÍCULAS

1. ARQUITETURA DO SISTEMA
O sistema adota uma arquitetura multicamadas baseada no padrão MVC (Model-View-Controller), composta por:
- Camada de Apresentação (View): Interface web desenvolvida com HTML e Thymeleaf, responsável pela interação com o usuário.
- Camada de Controle (Controller): Gerencia as requisições HTTP, valida dados e coordena o fluxo entre as camadas.
- Camada de Serviço (Service): Implementa regras de negócio e orquestra operações entre controladores e repositórios.
- Camada de Persistência (Repository): Realiza operações de acesso ao banco de dados via ORM (Spring Data JPA).
- Camada de Modelo (Model/Entity): Representa as entidades do domínio e o mapeamento objeto-relacional.
A comunicação entre módulos ocorre por injeção de dependências e passagem de DTOs. O padrão RESTful é utilizado para rotas e operações CRUD. Diagrama conceitual (texto):
[Usuário] <-> [View] <-> [Controller] <-> [Service] <-> [Repository] <-> [Banco de Dados]

2. MÓDULOS E FUNCIONALIDADES
- Aluno: Cadastro, edição, listagem e consulta de alunos.
- Curso: Gerenciamento de cursos, incluindo criação, edição e listagem.
- Matrícula: Realiza matrícula, trancamento, cancelamento, reativação e consulta de status.
- Relatórios: Geração de relatórios de alunos por curso e cursos mais procurados.
- Autenticação/Segurança: (Se implementado) Controle de acesso e permissões.
Cada módulo interage por meio de controladores, serviços e repositórios, garantindo separação de responsabilidades.

3. BANCO DE DADOS
O modelo de dados é relacional, com as principais tabelas:
- ALUNO (id, nome, email, ...)
- CURSO (id, nome, descricao, ...)
- MATRICULA (id, id_aluno, id_curso, numero_matricula, status, data_matricula, data_cancelamento)
Relacionamentos:
- ALUNO 1:N MATRICULA
- CURSO 1:N MATRICULA
Chaves primárias: id (todas as tabelas)
Chaves estrangeiras: id_aluno (MATRICULA -> ALUNO), id_curso (MATRICULA -> CURSO)
O modelo segue normalização até 3FN, garantindo integridade referencial. Tipos de dados: INT, VARCHAR, DATETIME. Triggers e procedures não foram identificados no contexto atual.

4. BACKEND E POO
O backend é desenvolvido em Java utilizando Spring Boot. A POO é aplicada por meio de:
- Classes: Entidades, DTOs, controladores, serviços e repositórios.
- Objetos: Instâncias das entidades e DTOs trafegam entre camadas.
- Herança: Possível uso em entidades ou serviços para especialização.
- Encapsulamento: Atributos privados e métodos públicos para acesso controlado.
- Polimorfismo: Interfaces (ex: MatriculaServico) e implementações (MatriculaServicoImpl).

5. CONTROLADORES, SERVIÇOS E DTOS
- Controladores: Recebem requisições, validam dados e invocam serviços.
- Serviços: Implementam lógica de negócio, manipulação de entidades e regras do sistema.
- DTOs: Transportam dados entre camadas, evitando exposição direta das entidades.
Exemplo: O controlador recebe um DTO, converte para entidade, chama o serviço, que persiste via repositório.

6. CONEXÃO COM FRAMEWORK E BANCO DE DADOS
- Framework: Spring Boot gerencia o ciclo de vida dos beans, rotas, injeção de dependências e configuração.
- ORM: Spring Data JPA realiza o mapeamento objeto-relacional, convertendo entidades Java em registros SQL.
- Repositórios: Interfaces que estendem JpaRepository, fornecendo métodos CRUD e consultas customizadas.
- Transações: Gerenciadas por anotação @Transactional, garantindo atomicidade das operações.
- Padrões de acesso: Repository Pattern, Service Layer.

7. OBSERVAÇÕES TÉCNICAS
- Padrões de codificação: Convenções Java, nomes significativos, separação de camadas.
- Boas práticas: Validação de dados, uso de DTOs, tratamento de exceções, logging com SLF4J/Logback.
- Segurança: (Se implementado) uso de Spring Security para autenticação/autorização.
- Tratamento de erros: Exceções customizadas, mensagens amigáveis ao usuário.
- Logs: Registro de operações relevantes e erros para auditoria e depuração.

8. TECNOLOGIAS UTILIZADAS
